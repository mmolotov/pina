== Executive Summary

*PINA* (Private Image Network Archive) is an open-source, self-hosted photo and video management platform.
It targets communities, families, and organizations that need a private, AI-powered media gallery with first-class group collaboration — something existing solutions do not adequately provide.

=== Vision

A self-hosted Google Photos alternative where groups of people can easily collect, organize, and search their shared photo and video libraries — with full control over their data.
Media and user data never leave the instance; optional outbound calls are limited to configured identity providers and model registries.

=== Key Differentiators

[cols="1,3"]
|===
| Differentiator | Description

| *Group onboarding*
| Admin posts an invite link in a chat — 100 people join a shared Space in seconds. No manual user management.

| *Hierarchical Spaces*
| Nested group structure (Space -> Subspace) that mirrors real-world organizations: clubs, families, teams.

| *Pluggable AI*
| Local-only ML inference with configurable pipeline. All models are free/open-source. No media data leaves the instance (models are downloaded from configured registries on first startup).

| *Telegram-native*
| Bot auto-collects photos and videos from group chats. Mini App provides a full client inside Telegram.

| *Multi-client*
| Web app, Telegram Mini App, Telegram Bot, and a versioned REST API for third-party integrations.
|===

=== Target Audience

* *Communities & clubs* — running clubs, hiking groups, sports teams with subgroups across cities
* *Families* — shared family photo & video archive with privacy control
* *Organizations* — event photography, corporate video, internal media libraries
* *Privacy-conscious individuals* — personal photo & video backup with local AI analysis

== Glossary

[cols="1,3"]
|===
| Term | Definition

| *Media Asset*
| An uploader-owned file (photo or video) identified within that uploader's library by its SHA-256 content hash. Stores the physical file, variants, and metadata. Exists independently of any album or Space. Different users may upload identical files as separate assets.

| *Media Reference*
| A link from an album or Space to a Media Asset. Adding a photo to a Space album creates a reference, not a copy. Removing a reference does not delete the asset.

| *Media Item*
| A generic term for any uploaded file — either a Photo or a Video.

| *Photo*
| An uploaded image file with associated metadata, variants, and ML analysis results.

| *Video*
| An uploaded video file with associated metadata, transcoded variants, poster thumbnail, and ML analysis results.

| *Variant*
| A derived version of a Media Item. For photos: original, compressed, thumbnails (SM / MD / LG). For videos: original, transcoded streams (multiple resolutions), poster image, animated thumbnail.

| *Album*
| A user-curated collection of Media References. Belongs to a Personal Library or a Space. Types: Personal, Space.

| *Favorites*
| A per-user virtual collection of bookmarked media items and albums. Each user maintains their own independent set of favorites.

| *Space*
| A collaborative group with its own albums and member roles. Can contain Subspaces. Media is added via references to assets owned by uploaders.

| *Subspace*
| A child Space nested under a parent Space. Inherits visibility rules by default; overrides are configurable per-subspace.

| *Personal Library*
| A private, per-user media collection. Separate from Spaces — has no roles, subspaces, or membership model. In the target auth flow it is created automatically on registration; Phase 1 already uses it as the ownership anchor for the active user context. Not visible to others unless explicitly shared.

| *Shared Link*
| A public, read-only URL granting access to any album or media selection without authentication. Sharing is a capability, not an album type.

| *Invite Link*
| A URL that allows a user to join a Space with a preconfigured role. In Phase 2 this works for existing authenticated users; new-user onboarding via invite remains a future extension.

| *ML Pipeline*
| A configurable sequence of AI analysis steps applied to each uploaded media item (photos directly, videos via extracted keyframes).

| *Model Registry*
| A catalog of available ML models with YAML manifests describing inputs, outputs, and capabilities.
|===

== Functional Requirements

Each requirement is marked with a phase indicator:

* *[MVP]* — required for the first usable release (Phase 1).
* *[Target]* — planned for Phases 2–4 (Auth, Spaces, Frontend, ML).
* *[Future]* — Phase 5+ (Telegram, Video, Advanced Features).
Design is directional, details may change.

=== FR-1: Photo Management [MVP]

==== FR-1.1: Photo Upload

[cols="1,4"]
|===
| ID | FR-1.1

| Description
a|
* *[MVP]* Users upload photos. The Media Asset belongs to the uploader and is created in the uploader's Personal Library.
* *[Target]* When Spaces are introduced (Phase 2): if a Space album is specified, a Media Reference is added automatically. The asset still remains in the uploader's Personal Library.

| Input
| Image file. Direct upload to a target album is [Target]. Supported formats: JPEG, PNG [MVP]; WebP, AVIF, HEIC, GIF [Target — requires ImageIO plugins].

| Processing
a|
. Validate file format and size limits (admin-configurable).
. Compute SHA-256 content hash.
. Deduplicate per uploader: if the same uploader uploads a photo with the same hash again, return the existing Media Asset [MVP]. Different users may upload identical files independently. When Spaces exist and a Space album is targeted, reuse the uploader-owned asset and create a new reference [Target].
. Store original file (if enabled by admin).
. Generate compressed version (configurable format, quality, max resolution).
. Generate three thumbnail variants:
* *SM* — square crop, 256x256px (for grid views).
* *MD* — proportional, max width 1280px (for previews).
* *LG* — proportional, max width 1920px (for full-screen viewing).
. Extract EXIF metadata: capture date, GPS coordinates, camera settings (exposure, aperture, ISO).
. Persist photo entity with all metadata and variant references.
. Dispatch asynchronous ML analysis task (when ML service is available) [Target].

| Output
| Photo metadata including ID, dimensions, EXIF data, and list of generated variants.

| Acceptance Criteria
a|
* Uploading a valid JPEG returns HTTP 201 with photo metadata.
* Uploading a duplicate (same SHA-256) by the same uploader returns the existing photo, no new file is stored.
* All three thumbnails are generated and retrievable.
* EXIF data (date, GPS, camera) is extracted and stored as JSON.
|===

==== FR-1.2: Photo Retrieval

[cols="1,4"]
|===
| ID | FR-1.2

| Description
| Users can browse their uploaded photos with pagination, retrieve photo metadata, download specific variants, and query geo-tagged photos for map-based browsing.

| Variants
| `ORIGINAL` (if stored), `COMPRESSED`, `THUMB_SM`, `THUMB_MD`, `THUMB_LG`.

| Behavior
| List endpoint returns a paginated response for the active user's uploaded photos. Metadata endpoint returns JSON with photo details and variant list. File endpoint streams the binary with correct Content-Type. Geo endpoints return the active user's geo-tagged photos for a bounding box or nearby-radius query, exclude photos without GPS coordinates, and use a marker-oriented payload for map browsing.

| Acceptance Criteria
a|
* `GET /photos?page=&size=&needsTotal=` returns the active user's uploaded photos in reverse chronological order with pagination metadata.
* `GET /photos/{id}` returns photo metadata with all variant info.
* `GET /photos/{id}/file?variant=THUMB_SM` returns the small thumbnail with `image/*` content type.
* [Implemented] `GET /photos/geo?swLat=&swLng=&neLat=&neLng=` returns only geo-tagged photos belonging to the active user for the requested viewport.
* [Implemented] Geo map payloads omit heavy fields for marker lists: `exifData = null` and `variants = []`.
* [Implemented] `GET /photos/geo/nearby?lat=&lng=&radiusKm=` returns only geo-tagged photos belonging to the active user within the requested radius and orders them by actual distance.
* Requesting a non-existent variant returns HTTP 404.
|===

==== FR-1.3: Photo Deletion

[cols="1,4"]
|===
| ID | FR-1.3

| Description
| Users can delete their own photos. Space admins/owners can delete any photo in their Space [Target].

| Behavior
a|
*[MVP]:*

* Deleting a photo with no album references removes the photo entity, all variants, and all files from storage.
* If the photo is still referenced by an album, deletion is rejected and files remain intact until the references are removed.
* No authorization checks yet — any user can delete any photo (Phase 2 adds ownership verification).

*[Target] — when Spaces are introduced:*

* Removing a photo from a Space album deletes the reference only — the asset remains in the uploader's Personal Library.
* Deleting a photo from the owner's Personal Library deletes the asset and all files from storage, but only if no other references exist. If references exist, the owner is warned.
* Space admins/owners can remove any reference from their Space.
* Non-owners receive HTTP 403 (unless Space admin/owner removing a reference).

| Acceptance Criteria
a|
* [MVP] Deleting a photo with no active album references removes all variant files from storage.
* [MVP] Deleting a photo with active album references returns a conflict and does not remove files.
* [Target] Removing a photo from a Space album does not delete files.
* [Target] An asset with active references cannot be silently deleted — the owner is prompted.
|===

=== FR-3: Spaces & Access Control [Target]

==== FR-3.1: Space Management

[cols="1,4"]
|===
| ID | FR-3.1

| Description
| Users can create Spaces for collaborative photo management. Spaces represent groups (clubs, families, teams).

| Attributes
| Name, description, avatar, visibility (public / private), creation date.

| Visibility
a|
* *Private* — not discoverable. Users can only join via invite link or direct invite.
* *Public* — listed in the Space catalog. Join-request moderation is a future extension; current Phase 2 access control is membership-based. Content is still only visible to members — public does not mean open access.

| Acceptance Criteria
a|
* Creating a Space makes the creator the Owner.
* Space has its own albums containing references to Media Assets owned by uploaders.
* Space can be renamed, described, and deleted by Owner.
* Deleting a Space removes all memberships, Space albums, and references. Media Assets remain in uploaders' Personal Libraries.
|===

==== FR-3.2: Subspace Hierarchy

[cols="1,4"]
|===
| ID | FR-3.2

| Description
| Spaces can contain Subspaces forming a tree hierarchy (e.g., "Runners" -> "Runners SPB" -> "SPB Beginners").

| Visibility Rules
a|
* [Implemented] Subspaces use adjacency list with `parent_id` + `depth` column. Maximum nesting depth is 5.
* [Implemented] Members of a *parent* Space can see albums in child Subspaces by default. This can be restricted per-subspace by an admin (`inheritMembers` flag).
* [Implemented] Members of a *child* Subspace can see only their level and below.
* [Implemented] Roles are inherited downward by default. A subspace admin can override inherited roles for specific members via direct membership.

| Acceptance Criteria
a|
* A Space can have multiple Subspaces.
* Subspaces can be nested up to depth 5.
* [Implemented] A user who is a Member of "Runners" can browse "Runners SPB" albums (when `inheritMembers=true`).
* [Implemented] A user who is only a Member of "Runners SPB" cannot see "Runners" root albums.
|===

==== FR-3.3: Roles & Permissions

[cols="1,4"]
|===
| ID | FR-3.3

| Description
| Each Space membership has a role that determines what the user can do within that Space.

| Roles
a|
[cols="1,4"]
!===
! Role ! Permissions

! *Owner*
! Full control: manage members, roles, settings, albums, photos. Delete Space. Transfer ownership.

! *Admin*

* An Admin can remove a Member but cannot remove the Owner.

=== FR-5: AI / ML Analysis [Target]

==== FR-5.1: ML Pipeline

[cols="1,4"]
|===
| ID | FR-5.1

| Description
| Each uploaded media item is processed by a configurable ML pipeline that runs asynchronously on the server. Photos are analyzed directly; videos are analyzed via extracted keyframes (see FR-11.5).

| Pipeline Steps (configurable, can be enabled/disabled by admin)
a|
* *Embedding generation* — CLIP-based vector embedding for semantic search.
* *Auto-tagging* — scene and object labels (e.g., "beach", "dog", "sunset").
* *Face detection* — bounding boxes for faces in the photo.
* *Face recognition* — identity matching against known faces in the Space/library.
* *Captioning* — natural language description of the photo.
* *OCR* — text extraction from images (signs, documents, whiteboards).
* *NSFW detection* — content safety classification.
* *Aesthetic scoring* — quality/beauty score for ranking and auto-curation.
* *Geo estimation* — approximate location from visual features (when no GPS in EXIF).
* *Duplicate detection* — perceptual hashing to find visually similar photos.

| Acceptance Criteria
a|
* Admin can enable/disable individual pipeline steps.
* Pipeline processes photos asynchronously (upload does not block on ML).
* ML results are stored in the database and associated with the photo.
* If the ML service is unavailable, photos are still uploaded successfully (ML runs later).
|===

==== FR-5.2: Model Registry

[cols="1,4"]
|===
| ID | FR-5.2

| Description
| ML models are managed through a registry with YAML manifests.

| Behavior
a|
* Each model has a manifest describing: name, version, task type, input/output format, download URL, license.
* Models are downloaded from HuggingFace on first startup or when a new model is configured.
* Admin can swap models for each pipeline step (e.g., replace CLIP ViT-B with ViT-L).
* Admin can upload custom ONNX models with a manifest.

| Constraints
| *All models must be free/open-source licensed.* No proprietary or cloud-only models.

| Acceptance Criteria
a|
* First startup downloads all configured models automatically.
* Admin can change a model and the system re-downloads if needed.
* Custom ONNX models can be uploaded and used in the pipeline.
|===

==== FR-5.3: Reindexing

[cols="1,4"]
|===
| ID | FR-5.3

| Description
| Admin can trigger re-analysis of existing photos with updated models or pipeline configuration.

| Options
a|
* *Full reindex* — re-process all photos.
* *Selective reindex* — by date range, album, Space, or specific pipeline step.
* *Priority* — newest first, favorites first, or by folder.
* *Background* — runs as a low-priority background task, does not block normal operations.

| Acceptance Criteria
a|
* Reindexing does not duplicate photos or variants.
* Progress is visible to admin (X of Y photos processed).
* Reindexing can be paused and resumed.
|===

=== FR-6: Search [Target]

==== FR-6.1: Text Search

[cols="1,4"]
|===
| ID | FR-6.1

| Description
| Users can search photos and videos by natural language queries.

| Behavior
| Query is converted to a CLIP embedding and compared against media embeddings using pgvector similarity search. Also searches auto-generated tags, captions, and video transcriptions.

| Examples

| Acceptance Criteria
a|
* Search returns results ranked by relevance, mixing photos and videos.
* Search respects access control (users only see media they have access to).
* Search works across Personal library and all accessible Spaces.
* Video results show poster thumbnail and duration badge.

a|

* Users can name a face cluster (e.g., "Alice").

==== FR-8.2: Storage Policies [Future]

[cols="1,4"]
|===
| ID | FR-8.2

| Description
| Admin can configure retention and compression policies to manage storage usage.

| Policies
a|
* *Store originals (photos)* — toggle per instance. When disabled, only compressed + thumbnails are kept.
* *Store originals (videos)* — toggle per instance. When disabled, only transcoded variants are kept after transcoding completes.
* *Photo compression format* — JPEG (default), WebP, AVIF (requires ImageIO plugin). Configurable quality (1-100) and max resolution.
* *Video transcoding profiles* — configurable resolutions and codecs (see FR-11.2).
* *Thumbnail sizes* — configurable dimensions for SM, MD, LG (applies to both photo and video poster thumbnails).
* *Retention* — auto-delete media older than N days from trash.
* *Quota* — per-user or per-Space storage limits (optional).

| Acceptance Criteria
a|
* Changing compression settings applies to new uploads only (existing photos unchanged unless reprocessed).
* Storage usage is visible per user and per Space.
* When quota is exceeded, uploads are rejected with a clear error message.

=== FR-9: Telegram Integration [Future]

==== FR-9.1: Telegram Bot

[cols="1,4"]
|===
| ID | FR-9.1

| Description
| A Telegram bot that auto-collects photos and videos from linked group chats into PINA.

| Features
a|
* *Link chat to Space* — admin connects a Telegram group to a PINA Space. All photos and videos posted in the group are automatically uploaded to the Space.
* *Album mapping* — configurable: all media to one album, or auto-create albums by date/topic.
* *Media types* — collects photos, documents (uncompressed photos), video messages, video files, and grouped media albums.
* *Commands*:
** `/browse` — list recent media (inline gallery).
** `/search <query>` — semantic search across accessible media.
** `/link <space>` — link current chat to a Space.
** `/unlink` — unlink current chat.
** `/status` — show link status and stats.

| Acceptance Criteria
a|

=== FR-11: Video Support [Future]

==== FR-11.1: Video Upload

[cols="1,4"]
|===
| ID | FR-11.1

| Description
| Users can upload videos with an optional target album (Personal or Space). The Media Asset is always created in the uploader's Personal Library. If a Space album is specified, a reference is added automatically.

| Input
| Video file (MP4, MOV, MKV, WebM, AVI), optional album target.

| Processing
a|
. Validate container format and size limits (admin-configurable max file size, default 2 GB).
. Compute SHA-256 content hash.
. Deduplicate per uploader: if the same uploader uploads a video with the same hash again, reuse the existing Media Asset. Different users may upload identical files independently.
. Store original file.
. Extract metadata: duration, resolution, frame rate, codec, audio codec, bitrate, creation date, GPS (if embedded).
. Generate poster image — a representative frame (configurable: first keyframe, frame at 25%, or auto-selected by aesthetic score).
. Generate animated thumbnail — short (3–5 second) looping preview (GIF or muted WebM), used in gallery grid on hover.
. Generate three poster thumbnails (SM / MD / LG) from the poster image, same as photo thumbnails.
. Dispatch transcoding task (asynchronous, see FR-11.2).
. Dispatch ML analysis task on extracted keyframes (asynchronous).

| Output
| Video metadata including ID, duration, resolution, codec info, and list of generated variants.

| Acceptance Criteria
a|
* Uploading a valid MP4 returns HTTP 201 with video metadata.
* Poster image and animated thumbnail are available immediately after upload completes.
* Video is playable via streaming even before transcoding finishes (if original format is browser-compatible).
|===

==== FR-11.2: Video Transcoding

[cols="1,4"]
|===
| ID | FR-11.2

| Description
| Uploaded videos are transcoded into web-friendly streaming formats asynchronously.

| Transcoding Profiles (admin-configurable)
a|
[cols="1,1,1,1"]
!===
! Profile ! Resolution ! Video Codec ! Use Case

! *360p*
! 640x360
! H.264
! Mobile on slow network

! *720p*
! 1280x720
! H.264
! Default playback

! *1080p*
! 1920x1080
! H.264
! High quality

! *Original resolution*
! Source
! H.265 / AV1 (optional)
! Archival, max quality
!===

| Behavior
a|
* Transcoding runs as a background task with configurable priority and concurrency.
* Each profile generates a separate variant stored via the Storage SPI.
* Admin can enable/disable individual profiles.
* If the original video is already in a browser-compatible format (H.264 MP4) and within resolution thresholds, transcoding for that profile is skipped.
* Hardware acceleration (VAAPI, NVENC, VideoToolbox) is used when available.

| Tooling
| FFmpeg (bundled in Docker image or system dependency).

| Acceptance Criteria
a|
* Transcoding happens asynchronously — upload returns immediately.
* Transcoding progress is queryable via API (percent complete).
* If transcoding fails, the original video remains accessible.
* Admin can re-trigger transcoding for a video.
|===

[cols="1,4"]

==== FR-11.4: Video Metadata

[cols="1,4"]
|===
| ID | FR-11.4

| Description
| The system extracts and stores comprehensive video metadata.

| Extracted Fields
a|
* *Duration* — total length in seconds.
* *Resolution* — width x height.
* *Frame rate* — FPS.
* *Video codec* — H.264, H.265, VP9, AV1, etc.
* *Audio codec* — AAC, Opus, etc. (or "no audio").

==== FR-11.5: Video in ML Pipeline

[cols="1,4"]
|===
| ID | FR-11.5

| Description
| Videos are analyzed by the ML pipeline using extracted keyframes.

| Behavior
a|
* Keyframes are extracted at configurable intervals (default: 1 frame per 5 seconds, plus scene changes).
* Each keyframe is processed through the photo ML pipeline (embedding, tagging, face detection, etc.).
* Results are aggregated at the video level:
** *Tags* — union of tags from all keyframes, weighted by frequency.
** *Faces* — all unique faces detected across keyframes, with timestamps of first/last appearance.
** *Embedding* — average of keyframe embeddings (for semantic search).
** *Caption* — generated from the most representative keyframe.
* Audio transcription (speech-to-text) is an optional pipeline step for searchable spoken content.

| Acceptance Criteria
a|
* Searching "birthday party" finds videos tagged from keyframe analysis.
* Face search returns videos containing the person, with approximate timestamp.
* ML analysis of videos does not block upload or transcoding.
|===

==== FR-11.6: Video Storage Policies

[cols="1,4"]
|===
| ID | FR-11.6

| Description
| Admin can configure video-specific storage and processing policies.

| Policies

* *Max upload size* — per-file limit (default: 2 GB, admin-configurable).
* *Transcoding profiles* — which resolutions to generate (enable/disable per profile).
* *Hardware acceleration* — enable/disable, select backend (VAAPI, NVENC, VideoToolbox, auto).
* *Transcoding concurrency* — max simultaneous transcoding tasks.
* *Animated thumbnail duration* — 3–10 seconds (default: 4).
* *Poster frame selection* — first keyframe, percentage, or aesthetic auto-select.

| Acceptance Criteria
a|
* Changing transcoding profiles applies to new uploads only (admin can trigger re-transcode for existing videos).
* Storage usage reports distinguish photo vs. video storage.

== Non-Functional Requirements

=== NFR-1: Privacy & Security

[cols="1,4"]
|===
| ID | NFR-1

| Requirement
a|
* No media data, user analytics, or telemetry leaves the instance. Outbound calls are permitted only to explicitly configured identity providers (OIDC) and model registries (HuggingFace).

* CORS is configurable per instance.

=== NFR-2: Performance

[cols="1,4"]
|===
| ID | NFR-2

| Requirement
a|
Benchmarks assume a mid-range self-hosted setup: 4-core CPU, 8 GB RAM, SSD storage, gigabit LAN.

* Photo upload response time < 3 seconds for a 10 MB JPEG (excluding ML processing).
* Video upload response time < 10 seconds for a 100 MB MP4 (excluding transcoding and ML processing).
* Gallery grid loads first page (50 thumbnails) in < 1 second on local network.
* Semantic search returns results in < 500ms for libraries up to 100,000 media items.
* ML pipeline and video transcoding process asynchronously — upload is never blocked.
* Thumbnail and poster delivery should leverage HTTP caching headers (ETag, Cache-Control).
* Video streaming supports HTTP range requests for instant seeking.

[cols="1,4"]
