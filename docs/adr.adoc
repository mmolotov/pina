Log of architectural decisions for the PINA project.
Each record follows a fixed structure: Status, Date, Context, Decision, Alternatives considered, Consequences, and Review trigger.

Statuses: *Accepted* (agreed, not yet in code) · *Implemented* (in code and tested) · *Planned* (intent declared, details open) · *Deprecated* · *Superseded*

== Language & Runtime

=== LANG-001: Java 25 as backend language

Status:: Implemented
Date:: 2026-03-25

==== Context

The backend needs a language with strong typing, mature tooling, and long-term vendor support.
The project targets self-hosters, so native-image compilation and low memory footprint matter.

==== Decision

Use Java 25 (LTS release, September 2025) with the Temurin distribution.
Only finalized language features are used (records, sealed interfaces, pattern matching, unnamed variables) — no preview flags.

==== Alternatives considered

- **Kotlin** — better syntax ergonomics but adds a compiler dependency and narrows the contributor pool.
- **Go** — fast builds, small binaries, but weaker ORM ecosystem and no equivalent to Quarkus Dev Services.

==== Consequences

- Access to the full JVM ecosystem (Quarkus, Hibernate, Flyway, Thumbnailator, metadata-extractor).
- Requires JDK 25 on CI and developer machines.

==== Review trigger

When the next Java LTS is released — evaluate migration.

=== LANG-002: Quarkus as application framework

Status:: Implemented
Date:: 2026-03-25

==== Context

The backend framework must support REST, CDI, ORM, DB migrations, health checks, and OpenAPI out of the box.
Fast startup and low memory are important for self-hosted deployments on modest hardware (Raspberry Pi, NAS, small VPS).

==== Decision

Use Quarkus with extensions: REST, Hibernate ORM Panache, Flyway, SmallRye OpenAPI, SmallRye Health, Dev Services.

==== Alternatives considered

- **Spring Boot** — larger community but higher memory footprint and slower startup.
No equivalent of Dev Services built-in.
- **Micronaut** — similar goals to Quarkus but smaller extension ecosystem, especially around Panache and Dev Services.

==== Consequences

- Hot reload via `quarkusDev` keeps the feedback loop tight.
- Dev Services auto-start PostgreSQL in dev/test mode — no manual DB setup needed.
- Ties the project to the Quarkus release cadence.

==== Review trigger

If Quarkus drops a critical extension or a competing framework offers a significant advantage for our use case.

=== LANG-003: TypeScript + React + Vite frontend

Status:: Implemented
Date:: 2026-03-25
Implementation notes:: Phase 3 SPA now includes the authenticated app shell, library, Spaces, favorites, settings, search shell, and URL-driven geo map browsing with client-side clustering.

==== Context

The frontend needs to handle media-heavy UIs: virtual scrolling, lazy image loading, gesture support, responsive grid layouts.

==== Decision

TypeScript SPA with React, React Router 7 (file-based routing), Vite, and Tailwind CSS.

==== Alternatives considered

- **Vue + Nuxt** — good DX but smaller ecosystem for media-heavy components.
- **Svelte + SvelteKit** — excellent performance but less mature library ecosystem for complex UIs.

==== Consequences

- Access to the largest ecosystem of media/gallery components.
- Vite provides instant HMR.
- Adds a Node.js dependency to the development toolchain.

==== Review trigger

If Phase 3/4 UI requirements add gesture-heavy viewers, large virtualized grids, or client-side performance bottlenecks — re-evaluate the stack and supporting libraries.

=== LANG-004: Separate Python ML service with gRPC

Status:: Accepted
Date:: 2026-03-25

==== Context

ML inference (tagging, face recognition, embeddings, captioning) requires Python-native model runtimes (ONNX, TensorRT).
The ML workload should scale independently from the API backend and crash without taking it down.

==== Decision

ML is a separate Python service.
Backend communicates with it over gRPC.
Proto definitions live in `proto/` and are shared between backend and ML.
FastAPI serves as the admin/health endpoint for the ML service.

==== Alternatives considered

- **REST for backend↔ML** — simpler, but gRPC is significantly more efficient for binary transfer of image data and embedding vectors.
- **Embedding ML in the backend (JVM)** — possible via ONNX Runtime Java bindings, but limits model ecosystem and complicates GPU management.

==== Consequences

- Independent scaling: ML workers can run on GPU machines while the backend runs on CPU.
- Adds operational complexity: two services to deploy and monitor.
- Proto definitions must be kept in sync.

==== Review trigger

When ML service implementation begins (Phase 4).

== Data Layer

=== DATA-001: PostgreSQL with pgvector

Status:: Implemented
Date:: 2026-03-25

==== Context

The application needs relational storage (users, albums, photos), JSONB (EXIF metadata), and vector similarity search (ML embeddings) — all in one database to keep self-hosted deployment simple.

==== Decision

PostgreSQL 17 is the sole database.
The pgvector extension is enabled from day one.

==== Alternatives considered

- **PostgreSQL + separate vector DB (Milvus, Qdrant)** — better vector performance at scale, but doubles operational burden for self-hosters.
- **SQLite** — zero-setup, but lacks pgvector, concurrent write support, and extensions.

==== Consequences

- Single database covers all storage needs.
- pgvector with HNSW indexes provides sufficient ANN search performance for personal/community-scale libraries (up to millions of photos).
- Self-hosters need only one database to back up and maintain.

==== Review trigger

If vector search latency becomes a bottleneck at scale — consider a dedicated vector store.

=== DATA-002: Flyway for schema management

Status:: Implemented
Date:: 2026-03-25

==== Context

Schema changes must be versioned, reviewable, and reproducible across environments.
The schema includes PostgreSQL-specific features (pgvector extension, custom types) that ORM auto-DDL cannot handle.

==== Decision

Database schema is managed exclusively by Flyway migrations.
Hibernate auto-DDL is disabled.
Migrations run at application startup (`migrate-at-start=true`).

==== Alternatives considered

- **Hibernate hbm2ddl** — convenient for prototyping, but cannot handle data migrations, index changes, or `CREATE EXTENSION vector`.
- **Liquibase** — more powerful than Flyway (rollback support), but added complexity is not justified at this stage.

==== Consequences

- Every schema change is a numbered SQL file in `db/migration/`.
- Rollbacks must be written manually if needed.

==== Review trigger

If rollback automation becomes critical — consider Liquibase.

=== DATA-003: Media Asset + Reference ownership model

Status:: Implemented
Date:: 2026-03-26

==== Context

The system needs a clear ownership model for media that works across Personal Libraries and shared Spaces.
Key scenarios: deduplication, deletion, Space removal, quotas, and access control all depend on who owns the physical file vs who can see it.

==== Decision

Uploader-owned asset + scoped references:

* *Media Asset* — an uploader-owned file identified by SHA-256 hash within that uploader's media library.
Physical files and variants belong to the asset.
* *Media Reference* — a link from a Space album to a Media Asset.
Adding media to a Space creates a reference, not a copy.
* Upload always creates the asset in the uploader's Personal Library.
If a Space album is specified, a reference is added automatically.
* Deduplication is per uploader (by uploader + content hash).
* Deleting a reference from a Space does not delete the asset.
* Deleting an asset from Personal Library deletes files only if no references exist elsewhere.

==== Alternatives considered

* **Per-scope copies** — simpler mental model, but wastes storage and breaks dedup across Spaces.
* **Global asset without Personal Library ownership** — complicates deletion (who can delete?), quotas (who pays for storage?), and leaks cross-user file existence.

==== Consequences

* Clear deletion semantics: reference removal is cheap, asset deletion is gated by reference count.
* Quotas can be attributed to the asset owner.
* Cross-user uploads of identical files do not leak prior existence and still preserve per-user ownership.
* Asset deletion is guarded by existing reference rows, without shared-blob reference counting across users.

==== Review trigger

[Reviewed 2026-03-28] Phase 2 (Spaces) validated — reference model handles concurrent delete (409 guard), Space deletion (DB cascade), and access-loss scenarios correctly. Next review: when implementing Video assets (Phase 7) — validate that videos reuse the same ownership model.

== Storage

=== STOR-001: Pluggable storage SPI

Status:: Implemented
Date:: 2026-03-25
Implementation notes:: Local FS provider is production-ready.
S3 and WebDAV are stubbed.

==== Context

Self-hosted users have different storage setups: local disk, NAS (WebDAV), S3-compatible object storage (MinIO, Backblaze B2).
The storage layer must be swappable without touching business logic.

==== Decision

`StorageProvider` interface with implementations selected at runtime via a CDI `@Produces` method and a config property (`pina.storage.provider=local|s3|webdav`).

==== Alternatives considered

- **CDI `@Alternative` / `@IfBuildProfile`** — locks the choice at build time, not at runtime.
- **Spring-style `@ConditionalOnProperty`** — Quarkus equivalent exists but CDI Producer is more idiomatic.

==== Consequences

- Adding a new storage backend = one class implementing `StorageProvider`.
- S3 and WebDAV implementations are stubbed, not yet production-ready.

==== Review trigger

Before S3/WebDAV providers are completed — validate the interface covers all required operations.

=== STOR-002: Photo-ID-based file storage paths

Status:: Implemented
Date:: 2026-03-25

==== Context

Users may upload the same photo multiple times from different devices, and different users may upload identical files independently.
The storage layout must stay simple while ensuring that one user's deletion cannot remove another user's files.

==== Decision

Files are named by `photo.id`.
SHA-256 remains the per-uploader deduplication key together with uploader identity, but it is not used as the physical filename.
Storage paths are date-partitioned: `{category}/{YYYY}/{MM}/{photo-id}.{ext}`.

==== Alternatives considered

- **Hash-based filenames** — look neat, but become unsafe once different users are allowed to store identical content without shared-blob lifecycle management.
- **Flat directory structure** — degrades on filesystems with >10k entries per directory (ext4, XFS).

==== Consequences

- Deduplication is automatic for repeated uploads by the same user.
- Different users get isolated physical files even when their assets share the same content hash.
- Date partitioning keeps directory sizes manageable and simplifies backup by time range.
- Concurrent upload of the same file by the same user is safe: the second insert fails with a constraint violation and returns the existing photo.

==== Review trigger

If a hash collision is ever observed (astronomically unlikely with SHA-256).

== API Design

=== API-001: REST + OpenAPI for client-facing APIs

Status:: Implemented
Date:: 2026-03-25

==== Context

The backend serves multiple clients: web frontend, Telegram Mini App, Telegram Bot, and potentially third-party integrations.

==== Decision

All client-facing endpoints live under `/api/v1/` with OpenAPI documentation auto-generated by SmallRye.
gRPC is reserved for the backend↔ML boundary.

==== Alternatives considered

- **GraphQL** — flexible for frontend queries, but adds complexity (schema stitching, N+1 protection) that is not justified for our API surface.
- **gRPC everywhere** — efficient, but poor browser support and harder to debug.

==== Consequences

- Swagger UI available at `/q/swagger-ui` for development and integration testing.
- URL-based versioning (`/v1/`) is explicit and easy to reason about.

==== Review trigger

If the API surface grows beyond ~30 endpoints and clients need flexible field selection — reconsider GraphQL.

== Infrastructure

=== INFRA-001: Monorepo

Status:: Implemented
Date:: 2026-03-25
Implementation notes:: CI with path-scoped triggers exists only for backend.
Other components do not have CI workflows yet.

==== Context

The project has six components (backend, frontend, ML, proto, tg-bot, tg-mini-app) that share API contracts and proto definitions.

==== Decision

All components live in a single Git repository with per-component build configs.
CI workflows are added per component as they mature, with path-scoped triggers to avoid unnecessary rebuilds.

==== Alternatives considered

- **Multi-repo** — stronger build isolation, but cross-component changes (e.g., proto update) require coordinated PRs across repos.

==== Consequences

- Atomic cross-component changes in a single PR.
- Single place for issues, documentation, and CI configuration.
- CI must use path filters to avoid rebuilding everything on every push.

==== Review trigger

If the team grows beyond 5 contributors and CI contention becomes a problem.

=== INFRA-002: Docker Compose as primary deployment target

Status:: Accepted
Date:: 2026-03-25

==== Context

The primary audience is self-hosters running PINA on a home server, NAS, or small VPS.
They need a single command to start the entire stack.

==== Decision

Docker Compose is the primary deployment method.
Kubernetes manifests may be added later but are not a prerequisite.

==== Alternatives considered

- **Kubernetes-first** — production-grade, but overkill for the target audience and raises the barrier to entry.
- **Bare-metal install scripts** — fragile across OS/distro variations.

==== Consequences

- Target state: `docker compose up` starts the entire stack (backend, frontend, ML, PostgreSQL).
Currently only backend + PostgreSQL are wired.
- Each component will have its own Dockerfile with multi-stage builds.

==== Review trigger

If a managed hosting offering or paid tier is considered — evaluate Kubernetes support.

== Authentication & Authorization

=== AUTH-001: JWT authentication with SmallRye JWT

Status:: Implemented
Date:: 2026-03-28

==== Context

The backend needs authentication that works for self-hosted deployments without requiring an external identity provider.
Multiple auth providers (local password, Google OIDC, Telegram Login Widget) will be supported over time.

==== Decision

Use SmallRye JWT for stateless token-based authentication.
Local registration uses username + BCrypt-hashed password.
JWT tokens contain the user UUID as subject claim.
A `linked_accounts` table links users to auth providers, allowing one user to have multiple login methods.

==== Alternatives considered

- **Session-based auth** — simpler, but requires sticky sessions or shared session store for horizontal scaling.
- **Quarkus OIDC only** — good for external providers, but self-hosted instances may not have an OIDC provider.

==== Consequences

- Stateless: no server-side session storage needed.
- RSA key pair required for signing and verification (included in resources for dev; must be externalized in production).
- BCrypt cost is configurable (`pina.auth.bcrypt.cost`): 12 for production, 4 for tests.
- `linked_accounts` pattern supports adding OIDC and Telegram auth without schema changes.

==== Review trigger

[Reviewed 2026-03-28] Google OIDC implemented — JWT claims and linked_accounts model confirmed sufficient. Next review: when adding Telegram Login Widget or GitHub OIDC.

=== AUTH-002: Linked accounts pattern for multi-provider identity

Status:: Implemented
Date:: 2026-03-28

==== Context

Users will authenticate via different providers (local password, Google, Telegram).
No single common field (email, phone) exists across all providers.
A user must be able to link multiple auth methods to one account.

==== Decision

Separate `linked_accounts` table with composite key (user_id, provider).
Each row stores provider-specific credentials (BCrypt hash for LOCAL, empty for OIDC, telegram_id for Telegram).

==== Alternatives considered

- **Single `users` table with email/password columns** — cannot represent users who only have Telegram auth.
- **Separate tables per provider** — redundant schema; harder to query all login methods for a user.

==== Consequences

- Adding a new provider requires only a new enum value in `AuthProvider`, not a schema change.
- Account linking logic must check for existing linked accounts to prevent duplicate identities.

==== Review trigger

[Reviewed 2026-03-28] Google OIDC implemented — pattern works as designed. Next review: when adding Telegram Login Widget.

=== AUTH-003: Adjacency list with depth for Space hierarchy

Status:: Implemented
Date:: 2026-03-28

==== Context

Spaces can have subspaces (e.g., "Trip 2026" → "Day 1", "Day 2").
The hierarchy needs simple queries (find parent, list children) and must prevent unbounded depth.

==== Decision

Adjacency list model: `spaces.parent_id` references `spaces.id`, plus a `depth` column.
Maximum depth is 5 (enforced in application and by DB CHECK constraint).
No re-parenting is supported in the initial implementation.

==== Alternatives considered

- **Nested sets** — efficient subtree queries, but expensive on insert/move and complex to maintain.
- **Materialized path** — easy to query full path, but requires path rewriting on move and string-based operations.
- **Closure table** — most flexible, but adds significant storage and insert overhead for a max-depth-5 hierarchy.

==== Consequences

- Simple queries: `WHERE parent_id = ?` for children, `WHERE id = parent_id` for parent.
- Depth column allows enforcing max depth without recursive queries.
- If re-parenting is needed later, cycle detection must be added (depth recalculation + ancestry check).

==== Review trigger

If re-parenting or deep subtree queries become a requirement.

=== AUTH-004: Google OIDC via jose4j direct verification

Status:: Implemented
Date:: 2026-03-28

==== Context

Google OIDC login requires verifying Google-issued ID tokens.
Quarkus provides a built-in OIDC extension, but it assumes the application acts as an OIDC client with redirect flows.
PINA receives ID tokens directly from the frontend (e.g., Google Sign-In button) and needs server-side verification only.

==== Decision

Use jose4j (already a transitive dependency of SmallRye JWT) to verify Google ID tokens directly against Google's JWKS endpoint.
`GoogleTokenVerifier` is a CDI bean that fetches Google's public keys via `HttpsJwks` and validates the token signature, issuer, audience, and expiry.
The Google client ID is configured via `pina.auth.google.client-id` (empty = disabled).

==== Alternatives considered

- **Quarkus OIDC extension** — designed for redirect-based flows; would require a different frontend integration pattern and adds unnecessary complexity for token-only verification.
- **Manual HTTP call to Google tokeninfo endpoint** — simpler, but adds a network round-trip per login and creates a dependency on Google's tokeninfo availability.

==== Consequences

- No additional dependencies needed (jose4j is already present).
- JWKS keys are cached and refreshed automatically by jose4j.
- Google OIDC can be disabled by not setting the client ID property — safe for self-hosted instances without Google credentials.
- Tests use `@InjectMock GoogleTokenVerifier` to avoid real Google calls.

==== Review trigger

If additional OIDC providers (GitHub, Apple) are added — consider extracting a generic OIDC token verifier.

=== AUTH-005: Subspace visibility via inheritMembers flag

Status:: Implemented
Date:: 2026-03-28

==== Context

Subspaces inherit members from parent Spaces by default, but some subspaces need restricted access (e.g., a leadership subspace within a public club).
The visibility control must be simple, per-subspace, and admin-manageable.

==== Decision

A boolean `inherit_members` column on the `spaces` table (default `true`).
When `true`, `getEffectiveRole()` walks up the parent chain and grants inherited roles.
When `false`, only direct memberships on that subspace are recognized — parent members cannot access it.
Direct memberships always work regardless of the flag.

==== Alternatives considered

- **Per-member visibility overrides** — more granular, but adds significant complexity (N rows per subspace instead of 1 flag).
- **Separate ACL table** — overkill for the current requirement; can be added later if needed.

==== Consequences

- Simple admin toggle: one boolean per subspace.
- Blocking at any level in the chain stops all inheritance below that point.
- Re-enabling the flag immediately restores access (no data loss).
- No extra ACL table is needed for Phase 2.

==== Review trigger

If per-member visibility overrides or more granular ACLs are needed.

=== AUTH-006: Invite link atomic join with usage tracking

Status:: Implemented
Date:: 2026-03-28

==== Context

Invite links have optional usage limits.
Multiple users may attempt to join via the same link concurrently.
The usage count must be accurate and must not exceed the limit under concurrent access.

==== Decision

The join operation uses pessimistic locking: `findByCodeForJoin()` loads the invite link with `LockModeType.PESSIMISTIC_WRITE` (`SELECT ... FOR UPDATE`), which serializes concurrent access to the same invite link row.
After acquiring the lock, the service validates the link (active, not expired, usage count below limit), increments the usage count, and creates the membership — all within the same transaction.
If validation fails, the link is invalid, expired, or exhausted — membership is not created.
Invite codes are 12-character alphanumeric strings generated with `SecureRandom`.

==== Alternatives considered

- **Atomic increment-first UPDATE** — single query combines validation and increment. Simpler, but harder to add conditional logic (e.g., skip increment for already-members).
- **Add member first, then increment** — creates a window where concurrent joins can both succeed before either increments, potentially exceeding the usage limit and requiring rollback.
- **SELECT + check + UPDATE (no lock)** — race condition: two concurrent joins could both pass the check before either increments the count.
- **Optimistic locking (@Version)** — requires retry logic on conflict.

==== Consequences

- `PESSIMISTIC_WRITE` serializes concurrent joins on the same invite link — no race conditions.
- No retry logic needed.
- Link exhaustion is immediate and accurate.
- Already-members can join without consuming usage count (the lock allows conditional logic before incrementing).
- Slightly reduced throughput under contention on a single invite link — acceptable for this use case.

==== Review trigger

If invite links need more complex join logic (e.g., admin approval queue).

=== AUTH-007: Refresh token flow with rotation

Status:: Implemented
Date:: 2026-03-28

==== Context

The original JWT setup used a single long-lived access token (24 hours).
Long-lived JWTs are risky: if stolen, they grant access until expiry with no revocation mechanism.
A refresh token flow allows short-lived access tokens while keeping sessions alive without re-authentication.

==== Decision

Short-lived access tokens (15 min) + long-lived refresh tokens (30 days) with single-use rotation.

* Access tokens are stateless JWTs signed with RSA. The backend never stores them.
* Refresh tokens are 32-byte random values (hex-encoded, 64 chars). The backend stores only the SHA-256 hash.
* Each call to `/auth/refresh` revokes the old refresh token and issues a new pair (access + refresh).
* `/auth/logout` revokes the refresh token. Access token remains valid until it expires (max 15 min).
* Refresh tokens are stored in a `refresh_tokens` table with `revoked` flag and `expires_at` timestamp.
* Both lifespans are configurable: `smallrye.jwt.new-token.lifespan` (access) and `pina.auth.refresh-token.lifespan` (refresh).

==== Alternatives considered

- **Long-lived access tokens only** — simpler, but no revocation and high risk if token is stolen.
- **Token blacklist for access tokens** — enables instant revocation, but requires a shared store (Redis) checked on every request, defeating the stateless benefit of JWTs.
- **Sliding-window refresh (no rotation)** — reuse the same refresh token until expiry. Simpler, but a stolen refresh token grants access for the full 30 days with no detection.

==== Consequences

- Stolen access tokens are limited to 15 min of damage.
- Refresh token rotation detects theft: if the attacker uses a rotated token, the legitimate user's next refresh fails, signaling compromise.
- No server-side state needed for access token validation (stateless JWT).
- Database lookup only on refresh/logout, not on every API call.
- JWT keys moved out of source control into `dev-keys/` (gitignored), configurable via env vars for production.

==== Review trigger

If real-time revocation of access tokens is needed (e.g., admin force-logout) — consider a token blacklist with Redis.

=== AUTH-008: Hybrid browser sessions alongside bearer-token auth

Status:: Implemented
Date:: 2026-04-03

==== Context

Bearer JWT + refresh-token rotation works well for API clients and future bot/integration flows, but web clients benefit from not exposing long-lived session material to JavaScript-accessible storage.
The frontend also needs a browser-native authentication mode that works with CSRF protection and can coexist with bearer-token auth during the transition to additional clients such as Telegram WebApp.

==== Decision

Keep bearer JWT + refresh tokens as the general API authentication mechanism, and add server-managed browser sessions for web clients.

* `POST /auth/session/register` and `POST /auth/session/login` issue a persistent browser session.
* The backend stores only hashed session and CSRF tokens in `browser_sessions`.
* The browser receives an httpOnly session cookie plus a readable CSRF cookie.
* Mutating browser-session requests must send `X-CSRF-Token`.
* `UserResolver` accepts either bearer JWTs or a valid browser-session principal.
* Session-issuing auth endpoints are rate-limited, and expired/revoked sessions are cleaned up asynchronously.
* Stored IP and user-agent hashes are audit metadata captured from trusted remote-address resolution;
  they are intentionally not used as hard session-binding checks.

==== Alternatives considered

- **Bearer tokens only for every client** — simpler surface area, but long-lived refresh material remains exposed to XSS in browser clients.
- **Session auth only** — would complicate non-browser clients and future bot/integration flows that are better served by explicit bearer credentials.
- **Opaque access tokens with server-side lookup on every request** — more centralized control, but loses the stateless benefits of JWT access tokens for API clients.

==== Consequences

- Web clients can authenticate without storing long-lived refresh material in persistent browser storage.
- CSRF protection becomes mandatory for mutating browser-session requests.
- Authentication stays hybrid: browser-session cookies for web UX, bearer tokens for programmatic clients.
- The backend now maintains server-side session state, cleanup jobs, and rate limiting for session issuance.
- Session provenance metadata remains useful for audit and future anomaly detection without causing
  false invalidations when IPs, proxies, or browser versions change.

==== Review trigger

When Telegram WebApp auth is implemented — verify whether it should issue the same browser session type or a dedicated session subtype with separate policy controls.

== Testing

=== TEST-001: Integration tests with real PostgreSQL

Status:: Implemented
Date:: 2026-03-25

==== Context

Database mocking hides SQL bugs, migration errors, and Hibernate mapping issues — the exact problems that break in production.

==== Decision

All tests run against a real PostgreSQL instance with pgvector via Quarkus Dev Services (Testcontainers).
No database mocking.

==== Alternatives considered

- **H2 in-memory database** — fast, but does not support pgvector, PostgreSQL-specific types, or advanced SQL features.
- **Mocked repositories** — fastest, but gives false confidence.
Prior incidents in other projects confirmed this risk.

==== Consequences

- Tests are slower but catch real issues.
- Docker is required on developer machines and CI.
- Flyway migrations are validated on every test run.

==== Review trigger

If test suite execution time exceeds 5 minutes — consider test parallelization or selective test profiles.
